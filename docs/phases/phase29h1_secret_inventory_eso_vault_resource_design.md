# Phase 29H.1 — Secret Inventory + ESO/Vault Resource Design

## Project

**SRE Decision Intelligence Platform**

## Phase status

**Enterprise design and implementation planning documentation**

Phase 29H.1 defines the company-grade secret management model for the SRE Decision Intelligence Platform before installing External Secrets Operator or deploying HashiCorp Vault resources.

The purpose of this phase is to design the secret architecture correctly before creating manifests.

This phase covers:

```text
secret inventory
secret ownership
Vault path design
External Secrets Operator design
Vault policy boundaries
Vault Kubernetes authentication model
ClusterSecretStore design
ExternalSecret design
GitOps repository impact
safe migration strategy
rotation model
validation commands
enterprise reasoning
```

---

## 1. Why Phase 29H.1 exists

Earlier Kubernetes phases introduced:

```text
PostgreSQL StatefulSet
API Deployment
Migration Job
Cilium Gateway
CiliumNetworkPolicy
Pod Security Admission
ResourceQuota
LimitRange
```

The platform now needs a company-grade approach for secrets.

The immediate problem:

```text
Kubernetes Secret manifests contain values such as:
- POSTGRES_PASSWORD
- DATABASE_URL
```

This is acceptable for temporary local testing, but not for GitOps.

In a company-grade GitOps environment:

```text
Git must not contain plaintext secret values.
Git should not contain encrypted secret material as the primary model.
Git should contain only secret references.
Vault should store the real secret values.
External Secrets Operator should sync approved values into Kubernetes.
```

---

## 2. Final enterprise decision

The primary secret management model for this project is:

```text
HashiCorp Vault + External Secrets Operator
```

Not:

```text
plaintext Kubernetes Secret YAML
```

Not primary:

```text
SOPS-encrypted secrets in Git
Sealed Secrets
```

SOPS and Sealed Secrets are valid tools in many teams, but for this project’s company-grade target, the stronger model is:

```text
Git stores ExternalSecret references only.
Vault stores real secret values.
ESO syncs Kubernetes Secrets from Vault.
Applications consume native Kubernetes Secrets.
```

---

## 3. Target architecture

```text
GitOps Repository
  stores only ExternalSecret manifests
        ↓
External Secrets Operator
  watches ExternalSecret resources
        ↓
ClusterSecretStore / SecretStore
  defines how to authenticate to Vault
        ↓
HashiCorp Vault
  stores real secret values
        ↓
Kubernetes Secret
  generated by ESO
        ↓
API / PostgreSQL / Migration Job
  consume native Kubernetes Secrets
```

This gives a clean separation of duties:

```text
GitOps controls desired state.
Vault controls secret values.
ESO controls synchronization.
Kubernetes workloads consume generated Secrets.
```

---

## 4. Why Vault + ESO is better for this target

### 4.1 SOPS model

```text
Git → encrypted secret file → Argo CD decrypts → Kubernetes Secret
```

This is useful, but encrypted secret material still lives in Git.

### 4.2 Vault + ESO model

```text
Git → ExternalSecret reference only
Vault → real secret values
ESO → Kubernetes Secret
Application → consumes Secret
```

This is stronger because:

```text
no plaintext secret values in Git
no encrypted secret values in Git as the primary model
centralized secret access control
centralized secret rotation
centralized auditability
clear separation between app teams and platform/security ownership
```

## 5. Enterprise decision table

| Area | Decision |
|---|---|
| Secret source of truth | HashiCorp Vault |
| Kubernetes sync mechanism | External Secrets Operator |
| GitOps repo stores | ExternalSecret references only |
| Runtime app consumes | Native Kubernetes Secret |
| Vault authentication | Kubernetes auth / workload identity |
| Secret rotation | Vault-first, ESO syncs update |
| SOPS | Fallback only |
| Sealed Secrets | Not primary |
| Plain Secret YAML in Git | Not allowed |

## 7. Secret ownership model

Every secret should have a clear owner.

| Secret area | Owner | Rotation responsibility |
|---|---|---|
| PostgreSQL credentials | Platform / Database owner | Platform team |
| API DB connection string | Platform team | Platform team |
| Registry pull secret | Platform / CI owner | CI/platform team |
| OpenSearch credentials | Observability owner | Observability/platform team |
| Prometheus token | Observability owner | Observability/platform team |
| API auth secrets | Security/platform owner | Security/platform team |
| Slack/webhook tokens | Incident management owner | SRE/platform team |

Secrets are not just YAML values. They are operational assets with lifecycle, ownership, rotation, and audit requirements.

## 12. External Secrets Operator design

External Secrets Operator will be installed in:

```text
external-secrets
```

The controller will:

```text
watch ExternalSecret resources
authenticate to Vault
read approved Vault paths
create/update Kubernetes Secrets
refresh secrets on interval
```

Recommended ESO controller identity:

```text
ServiceAccount:
  external-secrets/external-secrets
```

Vault authentication method:

```text
Kubernetes auth
```

Vault role:

```text
sre-decision-intelligence-eso
```

Vault policy:

```text
sre-decision-intelligence-read
```

---

## 13. Why Kubernetes auth instead of static Vault token

ESO should not authenticate to Vault using a long-lived static token.

Preferred:

```text
ESO ServiceAccount token
        ↓
Vault Kubernetes auth
        ↓
Vault role
        ↓
Vault policy
        ↓
read approved secret paths
```

Benefits:

```text
workload identity model
no long-lived Vault token in Kubernetes manifest
policy-bound access
namespace/serviceaccount scoping
clear audit trail
```

---

## 14. Vault policy design

Policy name:

```text
sre-decision-intelligence-read
```

Policy:

```hcl
path "secret/data/sre-decision-intelligence/*" {
  capabilities = ["read"]
}

path "secret/metadata/sre-decision-intelligence/*" {
  capabilities = ["read", "list"]
}
```

Why include metadata?

For KV v2, secret values are stored under:

```text
secret/data/...
```

Metadata lives under:

```text
secret/metadata/...
```

Some operations may need metadata read/list access.

Do not grant:

```text
create
update
delete
sudo
```

ESO should read secrets, not manage secret lifecycle.

## 15. Vault Kubernetes auth role design

Vault role:

```text
sre-decision-intelligence-eso
```

Bound ServiceAccount:

```text
external-secrets
```

Bound namespace:

```text
external-secrets
```

Attached policy:

```text
sre-decision-intelligence-read
```

Conceptual command:

```bash
vault write auth/kubernetes/role/sre-decision-intelligence-eso \
  bound_service_account_names=external-secrets \
  bound_service_account_namespaces=external-secrets \
  policies=sre-decision-intelligence-read \
  ttl=1h
```

This means only the ESO controller ServiceAccount in the `external-secrets` namespace can authenticate with this role.

## 16. ClusterSecretStore design

Use a `ClusterSecretStore` because the platform team can centrally define Vault connectivity.

Resource:

```text
ClusterSecretStore/vault-backend
```

Conceptual manifest:

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ClusterSecretStore
metadata:
  name: vault-backend
spec:
  provider:
    vault:
      server: "http://vault.vault:8200"
      path: "secret"
      version: "v2"
      auth:
        kubernetes:
          mountPath: "kubernetes"
          role: "sre-decision-intelligence-eso"
          serviceAccountRef:
            name: external-secrets
            namespace: external-secrets
```

Important:

```text
server value depends on how Vault is deployed.
```

Possible Vault service addresses:

```text
http://vault.vault:8200
http://vault.vault.svc.cluster.local:8200
https://vault.company.internal
```

Company-grade production target:

```text
Use TLS.
Prefer https://vault...
```

---

## 17. SecretStore vs ClusterSecretStore decision

| Store type | Scope | Use case |
|---|---|---|
| `SecretStore` | Namespace-scoped | App team owns its own backend config |
| `ClusterSecretStore` | Cluster-scoped | Platform team owns centralized backend config |

For this platform:

```text
Use ClusterSecretStore.
```

Reason:

```text
Vault connectivity is platform-owned.
Application namespaces consume approved secret paths through ExternalSecret.
```

## 18. PostgreSQL ExternalSecret design

Resource:

```text
ExternalSecret/sre-decision-postgres-secret
```

Target Secret created by ESO:

```text
Secret/sre-decision-postgres-secret
```

Conceptual manifest:

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: sre-decision-postgres-secret
  namespace: sre-decision-intelligence
  labels:
    app.kubernetes.io/name: sre-decision-postgres
    app.kubernetes.io/component: database-secret
    app.kubernetes.io/part-of: sre-decision-intelligence-platform
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: ClusterSecretStore
  target:
    name: sre-decision-postgres-secret
    creationPolicy: Owner
  data:
    - secretKey: POSTGRES_DB
      remoteRef:
        key: sre-decision-intelligence/postgres
        property: POSTGRES_DB
    - secretKey: POSTGRES_USER
      remoteRef:
        key: sre-decision-intelligence/postgres
        property: POSTGRES_USER
    - secretKey: POSTGRES_PASSWORD
      remoteRef:
        key: sre-decision-intelligence/postgres
        property: POSTGRES_PASSWORD
```

Note:

```text
Because ClusterSecretStore uses path: secret and version: v2,
remoteRef.key usually starts below the secret engine path.
```

So use:

```text
sre-decision-intelligence/postgres
```

not:

```text
secret/data/sre-decision-intelligence/postgres
```

## 21. ExternalSecret target creation policy

Important option:

```yaml
target:
  creationPolicy: Owner
```

Meaning:

```text
ESO owns the generated Kubernetes Secret.
```

During migration, if a Secret already exists, safer temporary option:

```yaml
target:
  creationPolicy: Merge
```

Migration recommendation:

```text
Use Merge temporarily if existing manual Secrets already exist.
Switch to Owner once ESO is the only source of truth.
```

Final GitOps target:

```text
creationPolicy: Owner
```

---

## 22. Safe migration strategy

Do not delete current Kubernetes Secrets first.

Safe sequence:

```text
1. Install External Secrets Operator.
2. Deploy or connect HashiCorp Vault.
3. Configure Vault Kubernetes auth.
4. Create Vault policy.
5. Create Vault role.
6. Put current secret values into Vault.
7. Apply ClusterSecretStore.
8. Apply ExternalSecret resources.
9. Confirm ESO creates or updates target Secrets.
10. Validate PostgreSQL.
11. Validate API /health/db.
12. Remove plaintext Secret manifests from Git.
```

This avoids breaking the platform during migration.

## 27. Validation without leaking secrets

Allowed:

```bash
kubectl get secret -n sre-decision-intelligence sre-decision-api-secret
kubectl describe externalsecret -n sre-decision-intelligence sre-decision-api-secret
```

Avoid:

```bash

kubectl get secret ... -o jsonpath='{.data.DATABASE_URL}' | base64 -d
echo $DATABASE_URL
```

If you must decode for private troubleshooting, do not paste it into public material.

## 28. Secret rotation model

Example: rotate PostgreSQL password.

```text
1. Generate new password.
2. Update Vault path:
   secret/data/sre-decision-intelligence/postgres
3. Update API Vault path:
   secret/data/sre-decision-intelligence/api
4. Update actual PostgreSQL user password.
5. Wait for ESO refresh or force refresh.
6. Restart API Deployment.
7. Validate /health/db.
8. Confirm old password no longer works.
```

Important:

```text
POSTGRES_PASSWORD and DATABASE_URL must stay consistent.
```

Later improvement:

```text
separate admin credential from application credential
automate DB credential rotation
use Vault database secrets engine for dynamic credentials
```

## 29. Future improvement: Vault Database Secrets Engine

A stronger enterprise model later:

```text
Vault Database Secrets Engine
```

Future flow:


This is more advanced and can be a later phase after the platform is stable.

## 31. CI/CD considerations

CI should not manage runtime secrets directly unless needed.

Rules:

```text
Do not print secret values.
Do not store Vault root token in CI.
Use OIDC or short-lived CI identity later.
Run secret scanning in CI.
Prevent plaintext secrets from entering Git.
```

This belongs later in:

```text
Phase 31 — CI Pipeline: Test, Build, Scan, SBOM, Sign, Push
```

OIDC is mainly used later for CI identity and API user authentication, while ESO uses Vault Kubernetes Auth first.
